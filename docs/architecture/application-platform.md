# Application Platform Layer

## Purpose
This document details the Application Platform Layer of the ATLAS system. It defines the public boundary between the ATLAS engine and external client adapters, establishing the rules for interacting with the platform.

## Responsibilities
- Serve as the single, canonical public interface for the ATLAS engineering platform.
- Isolate external clients from the internal complexities of the engine and domain models.
- Provide a stable, versionable API contract using the Command-Result pattern.
- Map internal engine exceptions to standardized application-level errors.

## Non-Responsibilities
- Implementing client-specific communication protocols (e.g., HTTP REST parsing, CLI argument parsing, MCP serialization).
- Executing core domain logic or modifying the filesystem directly.

---

## The Public SDK

The `atlas` package is the official Software Development Kit (SDK) for the platform. It is the **only** supported integration point for first-party and third-party clients.

### Clients
ATLAS itself is a local engineering platform, not an end-user application. All user interfaces are considered **Clients**. Examples of clients include:
- Command-Line Interfaces (CLI)
- Desktop Applications
- Web Applications
- Editor Integrations (VS Code, Cursor, JetBrains)
- MCP Servers
- REST APIs

**Rule**: No client may bypass the Application Layer. No client may import directly from `engine/` or call internal Workflow, AI, or Domain services directly.

## Architectural Patterns

### The Facade (`Atlas` class)
The primary entry point is the `Atlas` class, which acts as a Facade. It accepts Commands, delegates execution to the appropriate internal engine services, and returns Results.

### Command-Result Pattern
Communication across the Application boundary is strictly enforced using Data Transfer Objects (DTOs):
- **Commands**: Immutable, serializable, and transport-independent Pydantic DTOs that represent a requested action (e.g., `CreateProjectCommand`, `ExecuteStageCommand`).
- **Results**: Pure DTOs containing the outcome of an operation (e.g., `ProjectResult`, `ProposalResult`). Results never leak internal domain entities or repository instances to the client.

### Exception Mapping
Internal exceptions generated by the engine (e.g., `ProjectNotFoundException`, `WorkflowNotFoundException`) are caught by the Application Layer and explicitly mapped to public application errors (e.g., `ProjectNotFoundError`, `WorkflowNotReadyError`). There are no catch-all error handlers; mapping is explicit and exhaustive to guarantee a stable error contract.

### Composition Root (`_bootstrap.py`)
The platform is constructed via an internal composition root (`_bootstrap.py`) which instantiates repositories, wires dependencies, and assembles the engine services. 

**Rule**: The Bootstrap process is entirely hidden from clients. Clients interact with the initialized platform but never directly interact with the composition root logic.

The distributable package includes `atlas/`; clients obtain a fully wired instance
through `atlas.create()`. Initialization failures are exposed as `BootstrapError`.
Workflow status reports the readiness evaluation and its blocking issues rather than
assuming that a stage is ready.
